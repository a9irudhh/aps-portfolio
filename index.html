1. Optimizing Delivery Routes

Problem

  • Food gets cold if the delivery is late.
  • Customers get annoyed waiting.
  • Bad ratings for Zomato and the delivery guy.
  • People might not order again.
  • Main reason: Traffic, roadblocks, or the delivery guy taking a longer route. We need to get the food to the customer as fast as possible, every time.

Why fix this?

  • Happy customers = more orders.
  • Delivery guys can do more trips, earn more.
  • Less fuel wasted, saves money.
  • If we’re faster than Swiggy, we win.

How do we solve it?

  • Think of the city as a map with points (junctions) and lines (roads).
  • Each road has a time (not just distance) — depends on traffic, jams, etc.
  • We use live traffic data to update these times.
  • We want the route that takes the least time, not just the shortest distance.

What do we use?

  • Dijkstra's Algorithm: Finds the fastest way from A to B. Checks all possible routes, picks the best.
  • A* Algorithm: Like Dijkstra, but smarter. Uses a guess (straight-line distance) to speed things up.
  • We keep updating the map as traffic changes, so the route can change mid-delivery if needed.

How it works

  1. Draw the city as a graph (points and lines, with time for each road).
  2. When an order comes in, get the current location of the delivery guy and the customer.
  3. Run A* (or Dijkstra) to find the fastest route right now.
  4. If traffic changes a lot, re-calculate the route on the fly.

Performance (Time and Space Complexity)

Algorithm Time Complexity Space Complexity Notes
Dijkstra’s O((V + E) log V) O(V) Checks all possible routes
A* O((V + E) log V)
(usually faster in practice)		 O(V) Uses heuristics, avoids useless routes
  • For big cities, A* is better because it doesn’t waste time checking useless routes.
  • We can also save/calculate popular routes in advance to make things even faster.

2. Estimating Time of Arrival (ETA)

Problem

  • ETA isn’t just one thing — it’s a sum of steps, and each step can change.
  • Food prep time depends on the dish.
  • Delivery guy might take time to reach the restaurant.
  • Transit time depends on the route.

Why fix this?

  • People trust the ETA — if it’s wrong, they get mad.
  • Less customer support headaches about late food.
  • Delivery guys can plan better.
  • Business can track if ops are running on time.

How do we solve it?

  1. Break ETA into 3 parts: prep, pickup, transit.
  2. Each part is a quick lookup in a table or array.
  3. Add them up step by step (like a mini DP).

What do we use?

  • Static lookups (tables/arrays for prep, pickup, transit times).
  • Bottom-up DP-like accumulation for ETA calculation.

How it works

Example

  • prep_time['pizza'] = 15
  • pickup_time[1.2 km] = 6
  • transit_time['route_42'] = 12
  • ETA = 15 + 6 + 12 = 33 minutes

3. Assigning Delivery Partners Efficiently

Problem

  • Multiple orders and multiple delivery partners at the same time.
  • Each order gets exactly one partner, and each partner gets exactly one order.
  • We want the total cost (or time) across all assignments to be as low as possible.
  • This is the classic Assignment Problem, solved optimally by the Hungarian Algorithm.

Why fix this?

  • Ensures fair and efficient load distribution.
  • Reduces delivery delays.
  • Avoids overworking or underutilizing partners.
  • Improves customer and partner satisfaction.
  • Lowers fuel and time costs per delivery.

How do we solve it?

  • Model the problem as a cost matrix:
    • Rows → Orders
    • Columns → Delivery Partners
    • cost[i][j] → estimated cost or time if partner j is assigned to order i
  • Apply the Hungarian Algorithm to find the minimum-cost one-to-one assignment.

What do we use?

  • 2D Arrays (Cost Matrix) to store and process cost/time for each assignment possibility
  • Hungarian Algorithm to solve the Assignment Problem optimally
  • Graph Matching Principles (bipartite graphs)
  • Matrix Manipulation for reduction, covering, and optimization steps

How it works

  1. Row Reduction – Subtract smallest element in each row.
  2. Column Reduction – Subtract smallest element in each column.
  3. Cover all Zeros – Use minimum number of horizontal/vertical lines.
  4. Check – If lines = matrix size, proceed to assignment. Else, adjust uncovered elements and repeat.
  5. Final Assignment – Based on covered zeros in the adjusted matrix.

Example Cost Matrix

Partner A Partner B Partner C
9 2 7
6 4 3
5 8 1
  • Optimal assignment:
    • Order 1 → Partner B
    • Order 2 → Partner C
    • Order 3 → Partner A
  • Total minimum cost = 2 + 3 + 5 = 10 units

Performance (Time and Space Complexity)

Metric Value Description
Time Complexity O(n³) The Hungarian Algorithm runs in cubic time for an n × n cost matrix
Space Complexity O(n²) Required for storing the cost matrix and auxiliary matrices during computation
  • Efficient for up to hundreds of assignments in real-time batch processing.

4. Implementing Search Auto-Completion

Problem

As a user types into the search bar (e.g., “Bi...”), show real-time food items, restaurants, or cuisines that match the prefix — using efficient DSA-based techniques, without ML.

Why fix this?

  • Boosts user engagement and search accuracy.
  • Reduces typing effort, especially on mobile.
  • Increases conversion (orders) by surfacing popular choices early.
  • Enables personalization layers later (optional).

How do we solve it?

  • We use a Trie (Prefix Tree):
    • Each node represents a character.
    • Edges connect characters to form words.
    • At any node, we can efficiently explore all full words that begin with a prefix.

Time and Space Complexity

Metric Value Explanation
Insert Time O(N × L) Insert N words, each of avg length L
Prefix Search Time O(P + K) P = prefix length, K = number of completions
Space Complexity O(N × L × A) A = alphabet size (limited, e.g., 26 or 128)
  • Very efficient for real-time operations even with 10,000+ items.

5. Order Volume Analysis

Problem

Analyze how many orders are placed during specific time ranges (e.g., between 12 PM and 2 PM), with efficient support for frequent updates and queries.

Why fix this?

  • Identifies peak business hours across cities and zones.
  • Enables smart scheduling of resources (chefs, riders).
  • Detects unusual spikes or dips for operational decisions.
  • Supports heatmap-like data analysis for admin dashboards.

How do we solve it?

  • Divide the day into discrete time slots (e.g., every 15 minutes → 96 slots/day).
  • Track the number of orders per slot, and support:
    • update(t, +1) → Add 1 order to time slot t
    • query(t1, t2) → Total orders between slot t1 and t2
  • Use a Fenwick Tree to support both operations in O(log n) time.

Time and Space Complexity

Metric Value Description
Update Time O(log n) For inserting new order count in slot t
Query Time O(log n) For calculating sum of orders in a time range
Space Complexity O(n) For storing order counts per slot
  • Scales smoothly even with second-wise tracking (n = 86400 slots/day).

6. Order Pattern Analysis

Problem

Identify recurring sequences in users' food orders (e.g., someone often orders “Pizza → Coke → Brownie”) to uncover behavioral patterns.

Why fix this?

  • Reveals natural ordering habits (e.g., drinks always with certain meals).
  • Enables personalized combos, coupons, or homepage shortcuts.
  • Can be used to suggest reorders quickly.
  • Offers data-driven marketing without behavioral prediction models.

How do we solve it?

  • Treat each user’s sequence of past orders as a string or list of items.
  • Look for the longest common contiguous pattern shared between them.
  • Use Dynamic Programming to compute the Longest Common Substring between two sequences.

Time and Space Complexity

Metric Value Explanation
Time Complexity O(m × n) m and n are the lengths of the two order sequences
Space Complexity O(m × n) 2D DP table to track substring lengths
  • Optimizations possible to reduce space to O(min(m, n)) using rolling arrays.

7. Enhancing Delivery Partner Ratings

Problem

  • Delivery partners often receive low ratings due to delays or incorrect orders.
  • Ratings directly impact their morale and future assignments.
  • We need a system to improve ratings and ensure fairness.

Why fix this?

  • Higher ratings improve delivery partner satisfaction and retention.
  • Fair ratings ensure trust in the system.
  • Better ratings lead to improved customer satisfaction.

How do we solve it?

  • Introduce a feedback system for customers to rate specific aspects like timeliness and order accuracy.
  • Use weighted averages to calculate overall ratings.
  • Provide training and incentives for delivery partners to improve performance.

What do we use?

  • Feedback forms integrated into the app.
  • Algorithms to calculate weighted averages.
  • Data analytics to identify areas for improvement.

How it works

  1. Customers rate delivery partners after each order.
  2. Ratings are categorized into timeliness, accuracy, and overall satisfaction.
  3. Weighted averages are calculated to determine the final rating.
  4. Delivery partners receive feedback and training based on ratings.

Performance (Time and Space Complexity)

  • Time Complexity: O(n) for calculating averages, where n is the number of ratings.
  • Space Complexity: O(n) for storing ratings and feedback.
